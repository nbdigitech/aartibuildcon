  5. Which of the following is the complete set of relational algebra operations (i.e., Any of the other relational algebra operations can be expressed as a sequence of operations from this set)( Marks: 0.00 )
    1.   { σ, π, ∪, ⨝ }
    2.   { σ, π, ∪, – }
    3.   { σ, π, ∩, – }
    4.   { σ, π, ∪, ρ, -, × }
      Explanation:
      Complete set of operations:
      These are the fundamental operations of the relational algebra that are sufficient to express any relational algebra query.
      { σ, π, ∪, ρ, –, × } → Complete set of RA operations

      ∩ (Intersection) can be implemented by ‘–’ operator
      A ∩ B = A – (A – B)
      Join (⨝ ) can be implemented by × and σ operators
      A ⨝ B = σcondition (A × B)
      Division(÷) can be implemented by ×, – and π operators
      R ÷ S = (π(R-S))-(π(R-S)((S×π(R-S)(R))-R))


  6. Consider the following relational schemas:
    Student(Fname, Sid, sex, age), courseInfo(courseId, coursename, Hours, Instructor name) and Enroll(Sid, courseId, sem).
    Write the efficient relational algebra expression for the following query
    “Find the Fnames of students who have enrolled for course having id 100”( Marks: 0.00 )
    1.   πFname(σcourseId=100(Enroll⨝student))
    2.   πFname(σcourseId=100(Enroll⨝courseInfo⨝student))
    3.   πFname((σcourseId=100 (Enroll))⨝student)
      Explanation:
      Though a, b, c are the correct relational algebra expressions for the given query, c is the efficient one.
      ⇒ “a” is not efficient, since we are performing join first and then to the resultant table, we are applying selection, but here selection condition is only on enroll table. So, we can perform selection condition on enroll table and then do the join of the resultant table with student in an efficient way.
      ⇒ “b” is not efficient, since courseInfo relation is not needed to get the required info.
      ⇒ “c” is efficient relational algebra expression since we are performing selection on enroll first and then do the join of the resultant table with student relation to get student names who enrolled for course having id 100.
      ⇒ “d” is not correct option since we are projecting tuples having courseId as “100” instead of selecting rows having courseId “100”. So, it is not correct relational algebra query.
    4.   πFname(πcourseId(Enroll⨝student))

  8. Consider the relation employee(ssn,name, sex, Dno) and department(Dnumber, Dname, Mgr_ssn) with ssn and Dnumber as keys in employee and department schemas respectively. Here Dnumber in the department is a foreign key referring to the Dno in employee. What does the following relational calculus query produces? [Assume each department has a manager]
    {e.name | employee(e) ∧ (∀d)(¬department(d) ∨ d.Dnumber ≠ e.Dno ∨ (d.Mgr_ssn ≠ e.ssn) ∨ (e.sex = ‘male’))}( Marks: 0.00 )
    1.   Names of all male managers
    2.   Names of all female employees who are not managers.
    3.   Names of all male employees who are not managers.
    4.   all the above
      Explanation:
      Given TRC query can be written as
      {e.name | employee (e) ∧ (∀d)(¬(department(d) ∧ d.Dnumber = e.Dno ∧ (d.Mgr_ssn = e.ssn)) ∨ (e.sex = ‘male’))}
      ⇒ {e.name | employee (e) ∧ (∀d)((department(d) ∧ d.Dnumber = e.Dno ∧ d.Mgr_ssn = e.ssn)→ (e.sex = ‘male’))}
      For a given employee to display in output, we will check whether he/she is manager of any department. If she/he is, then we will display that employee name only he is male, otherwise we won’t display in output.
      If given employee is not manager of any department, we will display that employee name irrespective of his/her sex. So, given TRC query produces all employees who are male managers and all employees who are either male or female but not managers. So, the correct option is (d).
